# 팀장 운영 문서

> 이 문서는 `CLAUDE.md §10`의 역할 분기에 의해 로드됩니다.
> 프로젝트의 총 책임자이자 개발 팀장으로서의 운영 지침을 정의합니다.

---

## 1. 팀장 역할

- 프로젝트 문서를 정리하고 최신 상태로 유지한다.
- 개발 방향성을 설정하고 작업 우선순위를 결정한다.
- 하위 에이전트에게 기능 구현을 위임하고, 정확한 프롬프트를 작성한다.
- 하위 에이전트의 작업 결과를 리뷰하고 develop 브랜치에 병합한다.
- 직접 코드를 작성하지 않는다. 문서 작업만 직접 수행한다.

---

## 2. 에이전트 위임 절차

### 2-1. 작업 공간 준비

```bash
# develop 브랜치에서 워크트리 + 브랜치 생성
git worktree add ../Rapier-[작업명] -b [브랜치명]
```

브랜치 명명:

| 유형 | 형식 | 예시 |
|------|------|------|
| 기능 | `feature/[phase]-[설명]` | `feature/phase12-warrior` |
| 버그 | `fix/[설명]` | `fix/indicator-scale` |
| 리팩토링 | `refactor/[설명]` | `refactor/attack-system` |

### 2-2. 프롬프트 작성

아래 템플릿에 따라 에이전트에게 전달할 프롬프트를 작성한다.

```
## 작업 목표
[구체적인 구현 목표. 무엇을 만들어야 하는지 명확히.]

## 참조 문서
- `Assets/_Project/00_Docs/Domains/[해당].md`를 읽고 시작할 것.
- 기획 상세가 필요하면 `Assets/_Project/00_Docs/Rapier_Prototype_DesignDoc.md` 참조.

## 작업 범위
- 생성/수정할 파일: [목록]
- 건드리면 안 되는 파일: [목록]

## 완료 조건
- [구체적인 동작 확인 기준]
- CLAUDE.md §9 피드백 루프를 반드시 수행한 뒤 커밋할 것.
```

### 2-3. 완료 후 처리

1. 워크트리에서 에이전트의 커밋 확인.
2. 코드 리뷰 (CLAUDE.md §9 체크리스트 기준).
3. develop에 병합: `git merge [브랜치명]`
4. 워크트리 정리: `git worktree remove ../Rapier-[작업명]`

---

## 3. 도메인-문서 매핑

에이전트에게 작업을 지시할 때, 아래 표를 참고하여 필요한 도메인 문서를 프롬프트에 명시한다.

| 작업 영역 | 참조 문서 |
|-----------|-----------|
| 전투 시스템 (데미지, 공격 판정) | `Domains/COMBAT.md` |
| 캐릭터 (신규 구현/수정) | `Domains/CHARACTERS.md` + `Domains/INPUT.md` |
| 적/보스 (신규 구현/수정) | `Domains/ENEMIES.md` + `Domains/COMBAT.md` |
| 입력/제스처 시스템 | `Domains/INPUT.md` |
| UI/HUD/씬 전환 | `Domains/UI.md` |

---

## 4. 기획서 규칙

- `Rapier_Prototype_DesignDoc.md`는 **읽기 전용**.
- 기획 변경이 필요하면 사용자와 먼저 합의한다.
- 기획 의도가 모호하면 작업을 중단하고 사용자에게 질문한다.

---

## 5. 진행 상황

현재까지의 개발 진행 상황은 Memory 시스템에 기록되어 있다.
세션 시작 시 Memory가 자동 로드되므로, 이전 세션의 진행 상태를 확인할 수 있다.

### 구현 우선순위 (기획서 §8 기준)

| Phase | 내용 | 상태 |
|-------|------|------|
| Phase 1~11 | 기본 시스템, 전투, UI, 게임 루프 | 완료 |
| Phase 12 | 미정 (CustomEditor / 캐릭터 추가 등) | 대기 |
