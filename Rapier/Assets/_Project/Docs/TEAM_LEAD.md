# 팀장 운영 문서

> 이 문서는 `CLAUDE.md §10`의 역할 분기에 의해 로드됩니다.
> 프로젝트의 총 책임자이자 개발 팀장으로서의 운영 지침을 정의합니다.
> 자동 로드되는 메모리(`~/.claude/projects/.../memory/`) 와 함께 사용하세요.

---

## 1. 팀장 역할

- 프로젝트 문서를 정리하고 최신 상태로 유지한다.
- 개발 방향성을 설정하고 작업 우선순위를 결정한다.
- 하위 에이전트에게 기능 구현을 위임하고, 정확한 프롬프트를 작성한다.
- 하위 에이전트의 작업 결과를 리뷰하고 develop 브랜치에 병합한다.
- 직접 코드를 작성하지 않는다. 문서 작업만 직접 수행한다.

---

## 2. 에이전트 위임 절차

운영 규칙의 상세 체크리스트는 **`workflow_agent_delegation.md`** 메모리에 있다 (자동 로드됨). 본 절은 절차의 흐름만 정의한다.

### 2-1. 작업 공간 준비

```bash
# develop 브랜치에서 워크트리 + 브랜치 생성
git worktree add ../Rapier-[작업명] -b [브랜치명] develop
```

브랜치 명명:

| 유형 | 형식 | 예시 |
|------|------|------|
| 기능 | `feature/[phase]-[설명]` | `feature/phase12b-lobby` |
| 버그 | `fix/[설명]` | `fix/phase12a-input-blocking` |
| 리팩토링 | `refactor/[설명]` | `refactor/attack-system` |

워크트리 생성 직전, develop 에 미커밋 문서 변경이 있다면 먼저 develop 에 커밋해야 워크트리가 최신 문서를 포함한다.

### 2-2. 프롬프트 작성

위임 프롬프트는 **자체완결적**이어야 한다 (에이전트는 부모 대화를 보지 못함). 다음 항목을 모두 포함:

- **작업 환경**: 워크트리 절대 경로, 브랜치, 기준 커밋, "develop 워크트리(`C:\GitHub\Rapier`)는 절대 건드리지 말 것"
- **cd 정책**: "첫 Bash 호출에서 단 한 번 `cd "<워크트리 절대 경로>"` 실행. 이후 cd 재호출 금지. Bash cwd 는 한 세션 내 유지됨." 또는 git 은 `git -C <절대 경로>` 사용 권장.
- **작업 목표 (의도된 동작)**: 사용자 합의된 기획을 그대로 인용. 추측 금지.
- **참조 문서**: 절대 경로 + 읽을 순서. 보통 `CLAUDE.md` → 관련 `Domains/*.md` → `Rapier_Prototype_DesignDoc.md`.
- **구현 상태**: "현재 Rapier 만 구현됨. Warrior/Assassin/Ranger 클래스는 존재하지 않음." 동시에 "수정은 OCP 만족하도록 Base 레벨에서 일관 처리할 것."
- **작업 범위**: 수정 허용 폴더 + **수정 금지 폴더** (다른 12-X 작업 영역 충돌 방지)
- **검증 단계** (3 종):
  1. **잠금/플래그 짝 grep 검증** — 모든 `Begin*` / `LockXxx` / `SetXxx(true)` / `StartCoroutine` 에 대응하는 해제가 모든 종료 경로(정상/취소/사망/`OnDisable`)에 있는지 확인. 잠금↔해제 매핑 표를 보고에 포함.
  2. **시나리오 트레이스** (5~7개, 명시 + 자체 발굴 격려)
  3. **CLAUDE.md §9 피드백 루프** (컴파일/리뷰 체크리스트/자체 트레이스)
- **사용자 직접 플레이 테스트 인지**: "에이전트 코드 리뷰는 의도 검증의 마지막 단계가 아니다. 사용자가 워크트리를 Unity로 열어 직접 플레이하여 검증한다. 의도된 동작 명세를 정확히 따를 것."
- **커밋 가이드**: `[Phase 12-X] 한국어 설명` 형식. 본문에 근본 원인 + 수정 위치 + 검증 결과. **금지**: `git push`, `git merge`, `git worktree remove`, `--amend`, `--no-verify`, force push.
- **보고 형식**: 200~400자, (1) 수정 함수와 변경 요지 (2) 시나리오 트레이스 결과 (3) 잠금/해제 매핑 표 (4) 커밋 SHA.

### 2-3. 사용자 검증 (필수)

에이전트의 코드 리뷰 합격만으로 작업을 종결하지 않는다. 사용자가 워크트리를 Unity로 직접 열어 플레이 테스트해야 한다.

- 병렬 분기여도 사용자 테스트는 **순차적**이다 (Unity 한 번에 한 프로젝트만 효율적 — 동시에 여러 워크트리 오픈 비효율).
- 워크트리는 Library/ 미생성 상태 → 첫 오픈 시 시간 소요. 본격 분기 전에 Unity Hub 로 한 번 열어 사전 생성하면 이후 빠름 (선택).
- 사용자 보고 결과:
  - **OK** → §2-4 머지로
  - **버그** → §2-5 후속 위임으로

### 2-4. 머지 (사용자 OK)

```bash
git merge --ff-only [브랜치명]
```

- 기본은 fast-forward 머지로 single line history 유지.
- `--no-ff` 는 권한 deny 리스트에 있어 권한 요청. 머지 커밋이 의미 있을 때만 사용자 별도 확인 후 진행.
- 머지 후 **워크트리는 즉시 정리하지 않는다** — 후속 수정 가능성 때문 (Iterative Fix Cycle 정책, `feedback_iterative_fix_cycle.md` 참조).

### 2-5. 회귀 발견 시 후속 위임

사용자가 버그를 보고하면:

1. 같은 워크트리(아직 보존 중) + 같은 브랜치에 후속 커밋을 추가하는 방식으로 위임.
2. 새 위임 프롬프트는 회귀 증상 + 근본 원인 분석 + 정확한 수정 사항 명시. 추측 금지.
3. 단일 후속 커밋 (amend 금지). 이전 커밋과 별도 SHA.
4. 머지: `git merge --ff-only [브랜치명]` (fast-forward 가능)
5. 사용자 재검증 → OK 또는 추가 후속 위임 사이클 반복.

회귀가 발견되어도 자기 책망 금지. 이 사이클은 정상적인 게임 개발 과정이다.

### 2-6. 안정 확인 후 워크트리 정리

사용자가 충분히 안정 확인했다고 판단하면:

```bash
git worktree remove ../Rapier-[작업명]
git branch -d [브랜치명]   # 또는 -D (deny되어 있음, 사용자 확인)
```

`git worktree remove` 는 권한 deny 리스트에 있다 — 사용자 확인 후 진행.

## 4. 기획서 규칙

- `Rapier_Prototype_DesignDoc.md` 는 팀장이 직접 수정한다 (도메인 문서도 마찬가지).
- 기획 변경이 필요하면 사용자와 먼저 합의한다.
- 기획 의도가 모호하면 에이전트는 작업을 중단하고 사용자에게 질문한다 (이를 프롬프트에 명시).
